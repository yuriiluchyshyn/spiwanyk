# Steering: Не створювати скрипти для документації

## Правило

**НЕ СТВОРЮВАТИ** додаткові скрипти, файли або документи для підсумування роботи, якщо користувач явно не просить про це.

## Що НЕ робити

❌ **Не створювати:**
- `CHANGES-SUMMARY.md`
- `DEPLOYMENT-GUIDE.md` 
- `SUCCESS.md`
- `QUICK-FIX.md`
- Інші підсумкові документи
- Скрипти для документування процесу
- Файли з описом змін

❌ **Не писати в кінці:**
- "Створю файл з інструкціями..."
- "Давайте створимо документацію..."
- "Тепер створю підсумок..."

## Що робити замість цього

✅ **Правильний підхід:**
- Виконати завдання користувача
- Дати коротку відповідь про результат
- Якщо потрібні інструкції - написати їх в чаті
- Оновити існуючі README/документи якщо потрібно

## Винятки

Створювати документи ТІЛЬКИ якщо:
- Користувач явно просить: "створи документацію", "напиши інструкції в файл"
- Це частина основного функціоналу (README для проекту, API документація)
- Це конфігураційні файли які потрібні для роботи

## Приклад правильної поведінки

**Замість:**
> "Створю файл SUCCESS.md з інструкціями..."

**Правильно:**
> "Готово! Проект налаштований. Для запуску використовуйте `./run.sh`"

## Мета

Зберегти чистоту проекту та не засмічувати його зайвими файлами документації, які користувач не просив створювати.
